## 🧭 1. ТЕКУЩИЙ СТАТУС РАЗВЕРТЫВАНИЯ

Система развернута на боевом сервере Ubuntu (`45.76.212.160`) в режиме **Advisory Mode** (режим наблюдения и алертинга). Торговый контур (модуль отправки ордеров) намеренно изолирован.

*   **Рабочая директория на сервере:** `/opt/gektor`
*   **Исполняемый демон:** Systemd-служба `gektor.service` (управляет запуском `/opt/gektor/main.py`).
*   **Версия системы:** `3.0.0 "GEKTOR-STRIKE"` (полный переход на декларативное управление зависимостями через Poetry, удаление legacy-кода).

## 🏗 2. АРХИТЕКТУРА И ТОЧКИ ВХОДА

Кодовая база спроектирована по принципам **Clean DDD (Domain-Driven Design)**:

```
gektor/
├── main.py                          # Главная точка входа демона (Advisory Mode)
├── gektor_control.sh                # Управляющий скрипт (status/clean/restart)
├── pyproject.toml                   # Спецификация зависимостей Poetry
├── src/
│   ├── domain/                      # Чистая бизнес-логика (без внешних зависимостей)
│   │   ├── exit_protocol.py         # Логика инвалидации торговых гипотез
│   │   └── schrodinger_ledger.py    # Двухфазный учет лимитов и ордеров
│   ├── application/                 # Сценарии использования (Use Cases)
│   │   ├── orchestrator.py          # Основной торговый оркестратор (Production)
│   │   ├── exit_protocol_service.py # Фоновые наблюдатели за сигналами выхода
│   │   └── sentinel.py              # Стражи зависания каналов (Flatline/Blackout)
│   └── infrastructure/              # Инфраструктурный слой (БД, API, Сеть)
│       ├── database.py              # Менеджер подключений (SQLite, Redis WAL)
│       ├── telegram_notifier.py     # Асинхронный неблокирующий Telegram-клиент
│       └── gektor_l2/               # Движок L2 стакана (Shared Memory)
└── tests/                           # Интеграционные и юнит-тесты
```

### Точки запуска:
1.  **Advisory Mode (`main.py`):**
    Запускает облегченный класс `GektorRadarCore`. Он инициализирует неблокирующее подключение к SQLite и Redis, запускает телеграм-нотификатор и переходит в режим пассивного ожидания аномалий. Сетевые задержки Telegram вынесены во вспомогательный поток (очередь), защищая ядро от фризов.
2.  **Trading Monolith (`src/application/orchestrator.py`):**
    Содержит полноценный класс `GektorOrchestrator` для боевой торговли (WS-ингестия Bybit, мьютексы, риск-параметры). Будет активирован при переходе к фазе автоисполнения.

## ⚡ 3. ВНЕДРЕННЫЕ ИНЖЕНЕРНЫЕ РЕШЕНИЯ (HARDENING)

*   **Zero-Allocation L2 Engine:** Интегрирован кольцевой буфер на структурах `ctypes` (`SHMDeltaRingBuffer`) в `NdOrderBookStateMachine`. Это исключает аллокацию памяти в куче (heap allocation) на "горячем пути" обработки котировок.
*   **Fast-Forward Splicing:** Алгоритм выравнивания последовательностей в реальном времени при разрыве WebSocket-сессий. Он склеивает REST-снапшоты и буферизованные WS-дельты без разрыва непрерывности стакана.
*   **Hot-Reload (`SIGHUP`):** Перехват сигнала `SIGHUP` вызывает безопасную перезагрузку процесса «на лету» через `os.execv` без остановки системного демона.
*   **Outbox & Redis WAL:** Все торговые сигналы и алерты сначала пишутся во временную SQLite-базу данных (Write-Ahead Logging) и Redis, гарантируя доставку даже в случае сбоя сети.
*   **Flatline Sentinel:** Мониторинг «замерзания» котировок по отдельным монетам. При отсутствии тиков в течение 15 секунд система активирует протокол частичной слепоты (`PARTIAL BLINDNESS`) и шлет оповещение в Telegram.

## 🛠 5. УПРАВЛЕНИЕ И ЗАПУСК (RUNBOOK)

Для управления демоном на сервере используется оптимизированный bash-скрипт `gektor_control.sh`:

1.  **Обновление кода на сервере с автоматическим сбросом локальных веток:**
    ```bash
    git fetch origin && git reset --hard origin/main
    ```
2.  **Запуск/Перезапуск с мгновенной валидацией синтаксиса:**
    ```bash
    ./gektor_control.sh restart
    ```
    *(Валидация кода теперь исключает медленную компиляцию директории `venv/` и выполняется мгновенно).*
3.  **Очистка кэша, застрявших стаканов и разделяемой памяти (SHM):**
    ```bash
    ./gektor_control.sh clean
    ```
4.  **Мониторинг логов демона:**
    ```bash
    sudo journalctl -u gektor.service -f -n 50
    ```

## 🎯 6. ПЛАН СЛЕДУЮЩИХ ШАГОВ ДЛЯ РАЗРАБОТЧИКОВ

1.  **Интеграция Outbox-релея в Advisory Mode:** Подключить `TelegramRelayWorker` для вычитки отложенных сообщений из БД вместо прямого вызова `notify_manual`.

3.  **Нагрузочное тестирование L2-стакана:** Валидация задержек SHM-буфера под нагрузкой > 50 000 тиков/сек.
